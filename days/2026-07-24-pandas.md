# 📅 2026-07-24 · Day 14 — pandas

불리언 인덱싱에서 **`&`·`|`는 비교 연산자(`==`·`>=`·`<`)보다 우선순위가 높다.** 그래서 괄호를 빼면 파이썬은 사람이 읽는 순서와 **다르게** 묶는다 — 터지거나(그나마 다행), **에러 없이 조용히 틀린 행**을 통과시킨다.

## 오늘의 문제

### ① 괄호 없는 불리언 인덱싱 — "틀린 출력"이 아니라 "아예 터짐"

**문제 (코드)** — 요구사항: *EU 서버의 30레벨 이상 유저만.*

| nick | level | server |
|------|-------|--------|
| Nova | 45 | EU |
| Rex | 12 | KR |
| Kai | 30 | EU |
| Zed | 60 | NA |
| Ivy | 28 | EU |

```python
elite_eu = players[players['server'] == 'EU' & players['level'] >= 30]
```

**내 예측 (그대로)** —
> 1. 실행은된다. but, 출력 결과물에 nickname은 출력되지 않는다.
> 2. 30레벨 이상의 모든 유저가 출력된다.
> 3. EU서버의 유저를 뜻하는 문장을 먼저 거를수 있도록 소괄호() 를 써야하는데 안썼다.

**실제** — **3번(원인)은 정확히 맞혔다.** 그러나 1·2번(결과)이 반대였다: 실행 자체가 안 되고 `TypeError`로 터진다.

```
TypeError: Cannot perform 'rand_' with a dtyped [int64] array and scalar of type [bool]
```

`&`가 `==`·`>=`보다 우선순위가 높아 파이썬은 이렇게 묶는다:
```python
players['server'] == ('EU' & players['level']) >= 30
#                     └────────┬────────┘  여기를 가장 먼저 계산
```
`'EU'`(문자열) `&` `players['level']`(정수 Series)는 성립하지 않는 연산 → **필터는 시작조차 못 한다.** 그 부분만 떼어 실행해도 동일한 `TypeError`가 나는 것으로 확인.

순수 파이썬으로 본 우선순위:
```
3 == 3 & 1    -> False    # 파이썬: 3 == (3 & 1) → 3 == 1
(3 == 3) & 1  -> 1        # 사람이 읽는 순서로 묶으면 이 값
```

**교정 (1회 합격)** —
```python
elite_eu = players[(players['server'] == 'EU') & (players['level'] >= 30)]
```
→ `Nova(45,EU), Kai(30,EU)`. 예측(`남음 Nova·Kai / 빠짐 Rex·Zed·Ivy`)도 **정확히 일치**. 경계값 `Kai=30`은 `>= 30`이라 포함, `Ivy`는 EU지만 28이라 탈락 — 두 조건을 각각 굴려 판단함.

함정 fix 회피: 괄호를 친 뒤 `&` 대신 `and`를 쓰면 `ValueError: The truth value of a Series is ambiguous`로 또 터진다(`and`는 값 하나의 참·거짓을 묻는 연산이라 행이 여러 개인 Series엔 못 씀). 처음부터 `&`로 작성해 회피.

**교훈** — 코드를 "사람이 읽는 왼→오른쪽"이 아니라 **"파이썬이 계산하는 순서"**로 읽는다. 한 줄에 `&`·`|`가 비교 연산자와 섞이면 우선순위부터 의심한다.

## 직접 구현

**스펙** — 카테고리가 `book` 또는 `toy` 이면서 재고(`stock`)가 10개 미만인 상품을 `low_stock`에 담아 출력.

| name | category | stock |
|------|----------|-------|
| Novel A | book | 3 |
| Robot Kit | toy | 15 |
| Mug | kitchen | 2 |
| Puzzle | toy | 7 |
| Comic B | book | 40 |
| Pan | kitchen | 25 |

**내가 짠 코드 (원문 보존 · 3회 변천)** — 목표 예측은 처음부터 `['Novel A', 'Puzzle']`로 정확했다. 틀린 건 **그 의도를 코드로 옮기는 과정**뿐.

❌ 시도 1 — `or` 사용 + `stock` 조건 괄호 없음:
```python
low_stock = products[(products['category'] == 'book' or products['category'] == 'toy') & products['stock'] < 10]
```
→ `ValueError: The truth value of a Series is ambiguous.` `and`와 마찬가지로 **`or`도 Series엔 못 쓴다** — 원소별 OR는 `|`.

❌ 시도 2 — `|`로 고치고 개별 괄호도 쳤지만 **묶음 괄호가 없음**:
```python
low_stock = products[(products['category'] == 'book') | (products['category'] == 'toy') & (products['stock'] < 10)]
```
→ **에러 없이** `['Novel A', 'Puzzle', 'Comic B']`. `Comic B`는 재고 **40**인데 통과. `&`가 `|`보다 우선순위가 높아 파이썬이 이렇게 묶은 탓:
```python
(cat=='book') | ( (cat=='toy') & (stock<10) )   # = "book이면 재고 무관 통과, 또는 toy면서 재고<10"
```
스펙은 `(book 또는 toy)` **이면서** `재고<10`인데, 계산된 건 `book` **또는** `(toy이면서 재고<10)` — 다른 문장이다.

> 참고로 `or`만 `|`로 고치고 `stock` 괄호를 계속 빼면 **6개 전부 통과**한다(`(bool Series & int Series) < 10`이 모두 참). 에러가 없어 더 위험.

✅ 시도 3 — OR 묶음에 괄호를 하나 더:
```python
low_stock = products[((products['category'] == 'book') | (products['category'] == 'toy')) & (products['stock'] < 10)]
```

**실행/검증 결과** — ✅ `['Novel A', 'Puzzle']` — 스펙 정확 일치. `Comic B`(재고 40)·`Robot Kit`(재고 15)·kitchen 2건 모두 정상 배제.

**교훈** — 조건이 여러 개인 불리언 인덱싱의 3단 규칙: ⑴ **개별 조건마다 각각 괄호**, ⑵ 연결은 단어(`and`/`or`)가 아니라 **기호(`&`/`|`)**, ⑶ **`&`와 `|`를 섞으면 먼저 묶여야 할 쪽에 괄호를 하나 더**(`&`가 `|`보다 우선). 셋 중 하나만 빠져도 깨지는데, 빠지는 위치에 따라 **터지거나(1) 조용히 틀리거나(2)** 한다.

## 그날의 핵심
- **`&`·`|`는 `==`·`>=`·`<`보다, 그리고 `&`는 `|`보다 우선순위가 높다.** 불리언 인덱싱은 개별 조건 괄호 + 기호 연결 + 혼합 시 묶음 괄호, 3단이 다 필요하다.
- **`and`/`or`는 Series에 쓸 수 없다**(`ValueError: truth value is ambiguous`) — 값 하나의 참·거짓을 묻는 연산이라서. 행마다 각각 판단하려면 원소별 `&`/`|`.
- 진짜 위험은 에러가 아니라 **에러 없이 그럴듯한 답**이다. 시도 1은 터져서 즉시 알았지만, 시도 2는 조용히 `Comic B`(재고 40)를 통과시켰다 — 출력 행 수만 보고 "됐다"고 넘기면 못 잡는다. **스펙과 한 행씩 대조**해야 잡힌다.
