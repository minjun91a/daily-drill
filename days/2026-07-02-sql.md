# 📅 2026-07-02 · Day 4 — SQL

## 오늘의 문제 (Day 1 개념 · 힌트 없이 cold 재출제)

### 오답 회복 — JOIN 행 복제(fan-out)를 힌트 없이 재진단
- **문제 (검토한 코드)**:
  ```sql
  -- 게임별 총 결제 매출
  SELECT g.title, SUM(pu.amount) AS revenue
  FROM games g
  JOIN purchases pu ON pu.game_id = g.id
  JOIN platforms pl ON pl.game_id = g.id   -- 게임 1 : 플랫폼 N (SELECT엔 안 쓰임)
  GROUP BY g.title;
  ```
  데이터: **스타폴** 결제 2건(30000·50000) + 플랫폼 3개(PC·PS5·Switch) / **네온레이스** 결제 1건(20000) + 플랫폼 1개(PC). 실제 매출 = 스타폴 80000, 네온레이스 20000.
- **내 예측 (그대로)**:
  - 진단(힌트 없이 1차): "실제매출과 다르다. 그 이유는 platforms의 join으로 인해 결제내역의 중복된 game_id의 행이 같은 수만큼 늘어나기 때문에 각 행마다 계산이 되기 때문에." → ✅ **진단 cold 합격** (Day 1엔 같은 함정을 'WHERE 때문'으로 착각했던 것을 넘어섬)
  - 예측 숫자(처음엔 숫자 누락 → 보완): "스타폴 6행, 네온레이스 1행 총 7행. revenue 스타폴 240000, 네온레이스 20000." → ✅ 배수가 게임마다 다른 것까지(스타폴 ×3, 네온레이스 ×1) 정확.
- **실제**: `platforms`에 스타폴이 3줄 → 스타폴 결제 2행이 각각 3배 복제 = 6행, 네온레이스는 플랫폼 1줄 → 1행. 붙은 행은 **7행**이지만 `GROUP BY title`이 접어 **출력은 2행**. 그 7행이 `SUM` 안에서 더해져 스타폴 (30000+50000)×3 = **240000**(실제 80000의 3배), 네온레이스 20000×1 = **20000**(플랫폼이 1개라 우연히 정확). 부풀림 배수 = 게임별 플랫폼 수.
- **교정 (고친 코드)**:
  ```sql
  SELECT g.id, g.title, SUM(pu.amount) AS revenue
  FROM games g
  JOIN purchases pu ON pu.game_id = g.id   -- fan-out 원인인 platforms JOIN 제거
  GROUP BY g.id;
  ```
  교정 변천: ① platforms 제거는 바로 맞힘 — 근본 원인 제거, `SUM(DISTINCT)` 꼼수 회피(다른 결제가 우연히 금액 같으면 또 틀림). · ② 그러나 첫 제출은 `FROM game`(테이블명 오타) + `ON pu.game_id = g.game_id`(games엔 없는 컬럼) 참조 에러 2개 → 스스로 교정 검증해 `games`·`g.id`로 수정 → 합격.
- **교훈**: 집계 전 "이 JOIN이 행을 늘리나?"를 먼저 묻는다. SELECT에 안 쓰는 JOIN(platforms)이 조용히 `SUM`을 부풀린다. **부풀림 배수는 그룹마다 다를 수 있다**(스타폴 ×3, 네온레이스 ×1) — "전부 같은 배수로 틀리겠지"가 아니라 그룹별로 굴려야. 내가 짠 교정 코드도 참조(테이블·컬럼)가 실재하는지 다시 검증한다.
- **보충 — `GROUP BY g.id`인데 `g.title`이 SELECT에 있어도 OK?**: `g.id`가 PK → title이 함수적 종속이라 MySQL이 허용(Day 1 복습에서 다룬 개념). 다른 DB(Postgres)·엄격 모드 호환을 위한 안전 습관은 `GROUP BY g.id, g.title`.

## 복습 (힌트 없이 · 같은 개념 새 시나리오)

### 복습 — fan-out으로 COUNT 부풀기 (커뮤니티 글/댓글)
- **문제 (검토한 코드)**:
  ```sql
  SELECT u.name, COUNT(p.id) AS post_count
  FROM users u
  JOIN posts    p ON p.user_id = u.id
  JOIN comments c ON c.post_id = p.id
  GROUP BY u.name;
  ```
  데이터: 민수 글 2개(10·11), 지영 글 1개(12). 댓글 = 글10에 2개·글11에 1개·글12에 3개. 실제 글 수 = 민수 2, 지영 1.
- **내 예측 (그대로)**: "1. 민수 = 6, 지영 = 3. 이유는 comments까지 join되어 comments 행까지 출력되어 계산됐다. 2. 실제 작성 글 수와 다르다(이유 동일). 3. 교정 = comments JOIN 제거." → **진단 개념 ✅**(fan-out 정확)·**교정 ✅**. 단 **민수 숫자 오산**(6 → 실제 3).
- **실제**: comments JOIN이 각 글을 *그 글의 댓글 수만큼* 복제. 민수 = 글10(댓글2) + 글11(댓글1) = **3행** → `COUNT(p.id)` = 3(6 아님). 지영 = 글12(댓글3) = 3행 → 3. 실제 글 수(민수 2·지영 1)와 다름. "6"은 두 글 모두 댓글 3개씩으로 본 오산 — 글11은 댓글 1개뿐.
- **교정 (고친 코드)**:
  ```sql
  SELECT u.id, u.name, COUNT(p.id) AS post_count
  FROM users u
  JOIN posts p ON p.user_id = u.id   -- 안 쓰는 comments JOIN 제거 → 민수 2, 지영 1
  GROUP BY u.id;
  ```
  근본 원인(불필요 JOIN) 제거. `COUNT(DISTINCT p.id)`로도 가능하나 제거가 더 깨끗.
- **교훈**: fan-out 배수는 **그룹마다뿐 아니라 각 부모 행(각 글)마다도 다르다** — 민수 글10은 ×2, 글11은 ×1. "글마다 N개씩"으로 뭉뚱그리면 예측 숫자를 틀린다. 각 부모 행의 배수 = 그 행에 붙는 자식(댓글) 수.

## 직접 구현 (합격 후 · 검증 → 생산)

### 직접 구현 — fan-out 안 나게 집계 쿼리 백지에서 짜기
- **스펙**: 온라인 강의 플랫폼(`instructors`·`courses`·`enrollments`). 강사별 ① 이름 ② 개설 강좌 수 `course_count` ③ 총 등록 수 `enroll_count`를 **등록 수 많은 순**으로 출력하는 SQL을 **백지에서** 작성(고칠 코드 없음). 데이터: 김코딩(강좌 101·102, 등록 3건), 박데이터(강좌 103, 등록 4건). 정답 = 박데이터 `1|4`, 김코딩 `2|3`.
- **내가 짠 코드 (변천, 원문 보존)** — 뼈대(`FROM instructors i JOIN courses c ON c.instructor_id=i.id JOIN enrollments e ON e.course_id=c.id`)는 동일, 바뀐 부분만:
  - 1차: `count(c.id) AS course_count, SUM(e.courst_id, student_id) AS enroll_count` + `ORDER BY enroll_count ASC` → ❌ 실행 에러. `courst_id` 오타 + `SUM(a, b)` 인자 2개 불가 + **enroll_count는 더하기(SUM)가 아니라 세기(COUNT)** 개념 오류 + 정렬 ASC(스펙은 많은 순=DESC).
  - 2차: `count(c.id) ..., count(e.student_id) ...` + `DESC` → 실행됨. enroll_count·정렬 정답. 그러나 **course_count가 `4/3`으로 부풀음**(실제 `1/2`). enrollments JOIN이 강좌 행을 등록 수만큼 복제(fan-out).
  - 3차(오진단): `count(c.instructor)` → ❌ 컬럼 없음. "세는 **컬럼**을 바꾸면 된다"고 생각 → **실행 증거로 반증**: 컬럼을 `c.instructor_id`로 바꿔도 여전히 `4/3`. `count()`는 '행'을 센다.
  - 4차: `count(DISTINCT c.id) AS course_count` → ✅ **합격**.
- **실행/검증 결과** (sqlite3 실제 실행):
  ```
  id | name    | course_count | enroll_count
  2  | 박데이터 | 1            | 4
  1  | 김코딩   | 2            | 3
  ```
  스펙과 일치. fan-out은 course_count에만 났다 — 접힌 표의 **1행 = 1 등록**이라 등록 세기는 grain이 맞고, 강좌 세기는 103이 4번 겹쳐 초과됐던 것.
- **교훈**:
  - 검증에서 *알아본* fan-out을 직접 *짜다가* course_count에서 재현했다 → **인식 ≠ 생산**. 둘 다 훈련해야.
  - `count()`는 **행**을 센다. JOIN으로 복제된 행이 있으면 어느 컬럼을 세든 부풀음 → 세는 대상의 grain이 행 단위와 맞는지 확인. 안 맞으면 `COUNT(DISTINCT 식별자)` 또는 사전 집계.
  - **DISTINCT 안전성은 함수가 아니라 '무슨 컬럼에 거느냐'가 좌우**한다: **유일 id**에 걸면 가짜 복제만 제거(안전), **값/금액**에 걸면 우연히 같은 진짜 데이터를 삭제(위험). 실측: `payments(10000, 10000)` 두 건 → `SUM=20000`(진실) vs `SUM(DISTINCT)=10000`(1건 증발) / `COUNT(id)=COUNT(DISTINCT id)=2`(안전).

## 그날의 핵심
- **Day 1의 함정을 힌트 없이 넘었다**: 같은 JOIN 행 복제를 Day 1엔 'WHERE 때문'으로 착각 → Day 4엔 fan-out을 cold로 진단. 개념이 전이됨.
- 부풀림 배수는 그룹마다 다르다 — 한 게임만 보고 "×3"으로 일반화 금지, 그룹별로 굴리기.
- 붙은 행(7) ≠ 출력 행(2). GROUP BY가 접기 전 복제된 행이 `SUM`을 부풀린다.
- 내 교정 코드도 검증 대상: 방향(불필요한 JOIN 제거)이 맞아도 테이블명·컬럼 참조 오타면 안 돌아간다.
- **(직접 구현)** 검증에서 알아본 fan-out을 집계 쿼리 짜다 재현 → `count()`는 행을 센다. 복제 있으면 `COUNT(DISTINCT 식별자)`. DISTINCT는 유일 id엔 안전, 반복되는 값(금액)엔 위험(진짜 데이터 삭제).
- **(복습)** fan-out 배수는 그룹마다뿐 아니라 **각 부모 행마다도 다르다**(민수 글10 ×2, 글11 ×1). "글마다 N개씩"으로 뭉뚱그리면 예측 숫자를 틀린다.
