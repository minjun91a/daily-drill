# 📅 2026-07-20 · Day 10 — SQL

## 오늘의 문제 (`!=` / `NOT IN`은 NULL을 조용히 버린다 · 3치 논리)

- **문제 (검토한 코드)**: 게임 운영툴에서 **"핵(cheating)으로 밴당한 게 아닌 유저를 전부"**(정상 유저 + 핵 아닌 사유로 밴된 유저) 뽑으려는 쿼리.
  ```sql
  SELECT nickname
  FROM players
  WHERE ban_reason != 'cheating';
  ```
  데이터: **players** = Alpha(`cheating`)·Bravo(`abuse`)·Charlie(`NULL`)·Delta(`NULL`)·Echo(`spam`). `ban_reason`이 `NULL` = 제재 없는 정상 유저. 동료 의도 = Bravo·Charlie·Delta·Echo(4명).
- **내 예측 (그대로)**:
  > 1. Bravo, Echo
  > 2. 일치하지 않다.
  > 3. where 문장에서 cheating만 제외시켰지만, null값을 반환해주는 쿼리를 작성하지 않았기 때문에 null인 유저도 반환하지 않는다.
  → **결과·방향 정확**(Bravo·Echo 2명, 의도와 불일치). 원인의 **메커니즘만 정밀화 필요.**
- **실제** (sqlite3 실제 실행): `Bravo`, `Echo` 2명. 한 행씩 `ban_reason != 'cheating'`의 진리값:
  ```
  Alpha    ban=cheating  -> FALSE     (제외)
  Bravo    ban=abuse     -> TRUE      (통과)
  Charlie  ban=NULL      -> UNKNOWN   (제외 ←)
  Delta    ban=NULL      -> UNKNOWN   (제외 ←)
  Echo     ban=spam      -> TRUE      (통과)
  ```

### 메커니즘 정밀화 — "안 써서"가 아니라 "TRUE를 못 만들어서"
`NULL != 'cheating'`은 **FALSE가 아니라 UNKNOWN**이다. NULL은 "값을 모름"이라 `!=` 비교 자체가 참/거짓을 못 낸다. 그리고 `WHERE`는 **결과가 정확히 TRUE인 행만** 통과시킨다 → UNKNOWN도 버려진다. 즉 NULL을 *따로 요청 안 해서* 빠진 게 아니라, `!=` 비교가 NULL 앞에서 **TRUE를 못 만들어서** 걸러진 것. `=`·`!=`·`IN`·`NOT IN` 전부 같은 함정.

- **교정 (고친 코드) — 변천 원문 그대로**:

  **① 1차 — ❌ 문법 에러 (`near "null": syntax error`)**
  ```sql
  WHERE ban_reason not in null and != cheating
  ```
  세 오류 동시: (a) `cheating` 따옴표 없음(컬럼으로 해석), (b) `and != cheating`은 왼쪽 피연산자 없음(각 조건은 자기 컬럼 재명시 필요), (c) `not in null` — NULL을 **살려야** 하는데 `NOT IN`은 **빼는** 연산자라 방향이 거꾸로 + NULL은 `IN`/`NOT IN`으로 애초에 못 걸러짐(UNKNOWN). → **`NOT IN`을 "조건에 맞나?" 안 묻고 가져온 패턴 오적용.**

  **② 2차 — ❌ 문법 에러 (`near "is": syntax error`) · "null 연산자를 모르겠어"**
  ```sql
  WHERE ban_reason != 'cheating' or is null;
  ```
  실은 `is null`이 **찾던 그 연산자**였다 — 손이 먼저 씀. `IS NULL`도 `!=`처럼 **왼쪽에 검사할 컬럼**이 필요한데 그게 빠져 `is`에서 멈춤(1차의 (b)와 똑같은 규칙 재발).

  **③ 3차 — ✅ 합격**
  ```sql
  WHERE ban_reason != 'cheating' or ban_reason is null;
  ```
  **실행 결과**: `['Bravo', 'Charlie', 'Delta', 'Echo']` → 의도(4명)와 일치.

- **교훈**:
  - `!=`·`<>`·`IN`·`NOT IN`은 **NULL을 조용히 버린다**(NULL 비교 = UNKNOWN, `WHERE`는 TRUE만 통과). `!=` 쓸 때마다 "이 컬럼에 NULL 있으면?"을 먼저 물어야 정상 데이터가 안 사라진다.
  - NULL을 살리려면 `=` 계열이 전부 막혔으니 **`컬럼 IS NULL`**을 `OR`로 명시한다.
  - **모든 조건은 자기 컬럼을 다시 쓴다** — `!= 'x'`도, `IS NULL`도 왼쪽에 컬럼이 있어야 한다.

## 직접 구현 (합격 후 · 검증 → 생산)

### 직접 구현 — "A도 아니고 B도 아니다" + NULL 포함, 백지에서 짜기
**스펙**: 게임 길드 관리툴 `members`에서 **"제재·탈퇴 상태가 아닌 모든 멤버"의 `nickname`**을 출력. 제재·탈퇴 = `status`가 `'left'` **또는** `'banned'`. `status`가 `NULL`(신규·상태 미정) 멤버도 **반드시 포함**. (함정 경고 없이 백지 작성)

| id | nickname | status | 기대 |
|----|----------|--------|:---:|
| 1 | Nova  | active | ✅ 포함 |
| 2 | Orion | left   | ❌ 제외 |
| 3 | Pixel | NULL   | ✅ 포함 |
| 4 | Quill | banned | ❌ 제외 |
| 5 | Rune  | NULL   | ✅ 포함 |

기대 결과 = `Nova`, `Pixel`, `Rune` (3명).

**내 구현 변천 (원문 그대로)** — `NULL` 처리(`or status is null`)는 **처음부터 정확**, 가운데 연결(`OR`↔`AND`)만 바뀜:

**① 1차 — ❌ 5명 전부 나옴 (`Nova, Orion, Pixel, Quill, Rune`)**
```sql
select nickname
from members
where status != 'left' or status != 'banned' or status is null;
```
한 행씩 진리값:
```
Orion   left    | s!=left=FALSE  s!=banned=TRUE   | OR → 통과 ←?!
Quill   banned  | s!=left=TRUE   s!=banned=FALSE  | OR → 통과 ←?!
```
어떤 값이든 'left'·'banned' 둘 중 **하나만** 될 수 있으니 `!= 'left' OR != 'banned'`는 **모든 값에서 무조건 참** → 아무도 못 걸러냄. Orion은 `!= 'banned'`(TRUE)로, Quill은 `!= 'left'`(TRUE)로 새어나감. **De Morgan 오류**(NULL 처리는 백지에서 정확히 전이됨).

**② 2차 — ✅ 합격**
```sql
select nickname
from members
where status != 'left' and status != 'banned' or status is null;
```
**실행 결과**: `['Nova', 'Pixel', 'Rune']` → 스펙 일치.

**교훈**:
- **"A도 아니고 B도 아니다" = `!= A` AND `!= B`** (= `NOT IN (A, B)`를 풀어쓴 것). `OR`로 이으면 항상 참이 되어 아무것도 못 거른다. `NOT (A OR B)` = `NOT A AND NOT B`(De Morgan) — 부정을 나눠 넣을 때 연결도 뒤집힌다.
- `NOT IN ('left','banned')`은 `!= 'left' AND != 'banned'`와 같다 — 처음에 쓰려던 `NOT IN`은 방향만 맞으면 **AND의 축약**이었다.
- **보너스(우선순위)**: `A AND B OR C`는 SQL에서 `AND`가 `OR`보다 우선순위가 높아 `(A AND B) OR C`로 파싱된다 → 괄호 없이도 의도대로 동작. 단 AND/OR를 섞을 땐 **괄호로 의도를 못박는** 습관이 안전.

## 그날의 핵심
- **`!=`·`<>`·`IN`·`NOT IN`은 NULL을 조용히 버린다.** NULL 비교는 UNKNOWN, `WHERE`는 TRUE만 통과. `!=` 쓸 때마다 "NULL 있으면?"을 묻고, 살리려면 `컬럼 IS NULL`을 `OR`로 명시.
- **모든 조건은 자기 컬럼을 다시 쓴다** — `!= 'x'`도 `IS NULL`도 왼쪽에 컬럼 필요. (교정에서 2번 걸림)
- **(직접 구현) "A도 B도 아니다" = `!= AND !=` (= `NOT IN`).** `OR`로 이으면 무조건 참. De Morgan으로 부정을 나누면 연결도 뒤집힌다. AND는 OR보다 우선순위가 높다.
