# 📅 2026-07-21 · Day 13 — SQL

`LEFT JOIN`의 **오른쪽 테이블 컬럼을 `WHERE`로 필터하면 바깥 조인이 조용히 `INNER JOIN`으로 무너진다** — "없으면 0으로 남겨야 하는" 요구사항이 통째로 사라짐. 필터는 `WHERE`가 아니라 `ON`에.

## 오늘의 문제

### ① LEFT JOIN + WHERE = 고객이 통째로 증발

**문제 (코드)** — 요구사항: *모든 고객과 그들의 2026년 주문 건수. 2026년 주문이 없는 고객도 반드시 `0`으로 결과에 나와야 한다.*

`customers`: (1 Alice)(2 Bob)(3 Carol)(4 Dave) · `orders`:

| id | customer_id | amount | order_date |
|----|----|----|----|
| 101 | 1 | 50 | 2026-03-10 |
| 102 | 1 | 30 | 2025-12-20 |
| 103 | 2 | 20 | 2026-05-01 |
| 104 | 4 | 40 | 2025-06-15 |

(Carol은 주문 0건, Dave는 2025년 주문만.)

```sql
SELECT c.name, COUNT(o.id) AS order_cnt_2026
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE o.order_date >= '2026-01-01'
GROUP BY c.id, c.name;
```

**내 예측 (그대로)** —
> 1. Alice, Bob 의 건수만 출력된다.
> 2. 요구사항을 이행하지 않는다.
> 3. WHERE 문의 조건식이 잘못되었다. 2026-01-01 보다 크거나 같은 수치값만 출력하도록 조건이 걸려있다.

**실제** — 출력 `Alice 1, Bob 1` (Carol·Dave 증발). **출력 예측은 정확.** 하지만 원인 진단이 얕았다: 문제는 "조건식이 틀린 것"이 아니다 — `>= '2026-01-01'`은 요구사항이 원하는 필터다. 진짜 원인은 **오른쪽 테이블(`o.order_date`) 필터를 `WHERE`에 두면, `LEFT JOIN`이 살려둔 빈 행까지 사후에 다시 걸러져 `LEFT JOIN`이 `INNER JOIN`처럼 무너진다**는 것. 두 고객이 사라진 메커니즘도 서로 다르다(코칭 질문으로 스스로 구분):
- **Carol(주문 0건)**: `LEFT JOIN`이 만든 행의 `o.order_date`가 `NULL` → `NULL >= '2026-01-01'`은 `UNKNOWN` → `WHERE`가 버림 (Day 10 NULL 비교).
- **Dave(2025년 주문만)**: `order_date`가 `'2025-06-15'`로 실재하지만 조건 미달 → 버림.

**교정 (변천)** — 세 번 걸렸다.

❌ 제출 1 — 문법 에러 (`is null`에 컬럼이 없음):
```sql
WHERE o.order_date >= '2026-01-01' and is null
```
→ `near "is": syntax error`

❌ 제출 2 (해석) — 컬럼 채우고 `AND` → 0 rows:
```sql
WHERE o.order_date >= '2026-01-01' AND o.order_date IS NULL
```
→ 한 값이 동시에 "2026 이상"이면서 "NULL"일 수 없음 → 아무 행도 통과 못 함 (Day 10 AND/OR·De Morgan 함정).

❌ 제출 3 (해석) — `OR`로 바꿈 → Carol만 살고 Dave는 여전히 증발:
```sql
WHERE o.order_date >= '2026-01-01' OR o.order_date IS NULL
```
→ `Alice 1, Bob 1, Carol 0` — **Dave 여전히 없음.** `IS NULL`은 "주문 0건(Carol)"만 건지고, "2025년 주문만 있는(Dave)"는 못 건진다. B에서 스스로 구분한 두 원인 중 **하나만** 막는 "그럴듯한 오답 fix".

✅ 정답 — 필터를 `ON`으로 옮기고 `WHERE` 제거:
```sql
SELECT c.name, COUNT(o.id) AS order_cnt_2026
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id AND o.order_date >= '2026-01-01'
GROUP BY c.id, c.name;
```
→ `Alice 1, Bob 1, Carol 0, Dave 0`. `ON`은 "이 주문을 붙일지"만 정하고, 안 붙어도 `LEFT JOIN`이 고객을 `NULL` 행으로 남긴다 → `COUNT(o.id)=0`. `WHERE`처럼 합쳐진 행을 사후에 지우지 않으므로 고객이 살아남는다.

**교훈** — `LEFT JOIN`에서 오른쪽 테이블 컬럼을 `WHERE`로 필터하면 바깥 조인이 `INNER`로 무너진다. "짝이 없으면 0/NULL로 남겨야 하는" 요구사항이면 필터는 `WHERE`가 아니라 **`ON`에 `AND`로**. `WHERE`=합쳐진 결과를 사후에 거름(행 삭제), `ON`=무엇을 붙일지 결정(안 붙어도 왼쪽 행 보존).

### ② 교정 결과 예측 — 두 개념이 흔들림

**문제** — 정답 쿼리(위 ✅)의 출력을 4명 각각 예측하라.

**내 예측 (그대로)** —
> 이렇게 출력을 한다면, Alice = 2건, Bob = 1건, Carol = 1건, Dave = 1건 출력이 된다.

**실제** — `Alice 1, Bob 1, Carol 0, Dave 0`. 3개 틀림.
- **Alice 2→1**: `ON`의 `>= '2026-01-01'`은 Carol·Dave만 거른 게 아니라 **Alice 자신의 2025년 주문(102)도 안 붙게** 거른다. "2026만"이 *모든 고객*에 적용되므로 Alice도 2026년 것 1건.
- **Carol/Dave 1→0**: 짝 없는 `NULL` 행에서 **`COUNT(o.id)`는 NULL을 세지 않는다**(Day 7 `COUNT(*)`≠`COUNT(컬럼)`). 예측 "1,1"은 정확히 `COUNT(*)`가 냈을 틀린 답. `COUNT(o.id)`라서 0이 올바르게 나온 것.

추가 혼동 — "0인 고객도 출력돼야 한다"를 "count=1"로 옮김. **"행이 등장한다"(행의 존재)와 "count 칸 값"(그 칸 숫자)은 별개.** Carol은 `('Carol', 0)`으로 등장 → 등장(✓) + 값 0(✓), 모순이 아니라 동시 성립. 오히려 `1`이면 안 한 2026년 주문을 했다고 거짓말하는 틀린 결과.

**교훈** — 코드를 맞게 써도 "무엇을 출력하는지"를 못 맞히면 검증이 안 된 것. ⑴ `ON` 조건은 조인 양쪽 모두에 적용된다. ⑵ `COUNT(컬럼)`은 NULL을 세지 않는다. ⑶ 요구사항의 "0으로 나온다"="행 등장 + 값 0".

## 직접 구현

**스펙** — 모든 플레이어와 그들이 2026년에 획득한 도전과제 수. 2026년에 하나도 없어도 반드시 `0`으로 등장. 컬럼 `nickname, ach_2026`, `player.id` 오름차순.

데이터: `players`(Nova/Rex/Kai/Zed) · `achievements`(Nova 2026×2 + 2025×1, Rex 2026×1, Kai 2025×1, Zed 없음).

**내가 짠 코드 (원문 보존)** —
```sql
SELECT
    p.nickname,
    count(a.id) AS ach_2026
FROM players p
LEFT JOIN achievements a ON a.player_id = p.id AND a.earned_date >= '2026-01-01'
GROUP BY p.id;
```

**실행/검증 결과** — ✅ `Nova 2, Rex 1, Kai 0, Zed 0` — 스펙 정확 일치. 배운 함정 전부 회피: 필터를 `ON`에 둬 Kai·Zed 보존, Nova의 2025 과제 제외해 2, `COUNT(a.id)`로 NULL 행 0 처리, id로 GROUP BY. **인식(오답 알아보기) → 생산(백지 작성) 전이 성공.**

보완 노트(정답이지만 실무 다듬기):
- **`ORDER BY p.id` 누락** — 스펙이 정렬을 요구했다. 결과는 우연히 id순으로 나왔지만 **`GROUP BY`는 행 순서를 보장하지 않는다.** 정렬이 스펙이면 `ORDER BY`를 명시.
- **`GROUP BY p.id`로 묶고 `p.nickname` SELECT** — SQLite는 허용. Postgres도 `p.id`가 PRIMARY KEY면 함수 종속으로 허용, 아니면 에러. 이식성·안전엔 `GROUP BY p.id, p.nickname`. (id로 묶은 것 자체는 옳음 — 동명 방지, Day 1.)

**교훈** — 다음엔 스펙의 정렬 요구까지 `ORDER BY`로 못박는다. 집계 컬럼 외 SELECT 컬럼은 GROUP BY에 함께 넣어 이식성 확보.

## 그날의 핵심
- **`LEFT JOIN` + 오른쪽 테이블 `WHERE` 필터 = 바깥 조인이 `INNER`로 무너짐.** "짝 없으면 0/NULL"이면 필터는 `ON`에. `WHERE`=사후 행 삭제, `ON`=붙일지 결정(왼쪽 행 보존).
- "그럴듯한 오답 fix"(`WHERE ... OR 오른쪽.col IS NULL`)는 짝이 아예 없는 경우(NULL)만 살리고, 짝은 있으나 조건 미달인 경우(2025년 주문만)는 못 살린다 — 증상만 땜빵.
- `COUNT(컬럼)`은 NULL을 세지 않는다(Day 7) · "행 등장"과 "count 값"은 별개 · 정렬 스펙은 `ORDER BY`로 못박기(`GROUP BY`는 순서 미보장).
