# 📅 2026-07-09 · Day 8 — pandas

## 오늘의 문제 (Series 산술 · 라벨 정렬 + 미매칭)

- **문제 (검토한 코드)**:
  ```python
  q1 = pd.Series({'서울': 120, '부산': 90, '대구': 60})
  q2 = pd.Series({'부산': 100, '대구': 80, '광주': 50})  # 다른 담당자 파일 — 지점 구성·순서 다름
  total = q1 + q2
  print(total)
  ```
  동료 주장: "`q1 + q2`면 지점별 상반기 합산 매출이 나온다."
- **내 예측 (그대로)**:
  > 1. 서울:120, 부산:190, 대구:140, 광주:50
  > 2. 인덱스는 안맞지만 각 딕셔너리의 키값으로 통합되서 결국 토탈값이 제대로 반환되지않나?
  > 3. 아무래도 내가 개념을 잘못생각하고 있을 수 있지만 딕셔너리로 묶여있기 때문에 각 키값되 벨류값이 인덱스 순서가 맞지 않아도 출력될테고, q1과 q2에 중복값이 아닌 키와벨류값들은 그대로 끌어와지지않을까 생각한다.

  → **겹치는 라벨(부산 190·대구 140)은 정확.** 그러나 **한쪽에만 있는 서울·광주를 "그대로 끌어온다"가 어긋남.**
- **실제** (python 실제 실행):
  ```
  광주      NaN
  대구    140.0
  부산    190.0
  서울      NaN
  dtype: float64
  ```

### 어긋난 곳 — 미매칭 라벨은 "통과"가 아니라 NaN

`q1 + q2`는 딕셔너리 병합(`update`)이 아니다. **두 인덱스의 합집합(union)에 라벨별로 정렬한 뒤 더한다.** 라벨 하나씩 손으로 굴리면:

- **부산**: 둘 다 있음 → `90 + 100 = 190` ✅
- **대구**: `60 + 80 = 140` ✅
- **서울**: q1엔 120, **q2엔 없음** → 없는 쪽을 `NaN`으로 채워 `120 + NaN` → **`NaN`**
- **광주**: q1엔 없음 → `NaN + 50` → **`NaN`**

핵심 2가지:
1. **한쪽에만 있는 라벨은 "그대로 통과"가 아니라 상대가 `NaN`으로 채워지고, `NaN`은 어떤 숫자와 더해도 `NaN`(전염).** → 예측 3번의 착각.
2. `NaN`이 섞이면 정수로 못 담아 **dtype이 `float64`로 승격** → 그래서 `140.0`. 동료 주장은 틀렸다: 서울·광주 매출이 조용히 증발해 `.sum()`이 두 지점을 통째로 빠뜨린다.

- **교정 (고친 코드)** — 변천:

  **① 1차 — ❌ TypeError + 마지막 줄이 버그로 되돌림**
  ```python
  q1.add(q2, '광주'>=0)   # '광주' >= 0 → TypeError (str vs int), 인자 계산에서 죽음
  total = q1 + q2          # 설령 위가 됐어도 이 줄이 다시 NaN 버전으로 덮어씀 + .add 반환값 버려짐
  print(total)
  ```
  세 오해가 겹침: (a) `fill_value`는 조건·라벨이름이 아니라 **없는 라벨을 채울 숫자 하나**인데 `'광주'>=0` 비교식을 넣음, (b) `.add()`는 원본을 안 바꾸고 **새 Series를 반환** → 받아야 함, (c) 마지막 `q1 + q2`는 버리려던 버그 코드.

  **② 2차 — ✅ 합격**
  ```python
  total = q1.add(q2, fill_value=0)
  print(total)
  ```
  **실행 결과**
  ```
  광주     50.0
  대구    140.0
  부산    190.0
  서울    120.0
  dtype: float64
  ```
  없는 쪽을 **더하기 전에** 0으로 채워 `120 + 0 = 120`, `0 + 50 = 50` → 값 보존.

- **③ 교정 검증 — 함정 fix `(q1+q2).fillna(0)`**:
  ```
  add(fill_value=0):    서울 120, 광주 50   ✅
  (q1+q2).fillna(0):    서울   0, 광주  0   ❌  ← 진짜 값이 날아감
  ```
  `.fillna(0)`은 먼저 `120 + NaN = NaN`으로 **원래 값을 지운 뒤** 0으로 덮는다. "0으로 채운다"도 **언제 채우냐**가 결과를 가른다 — 없는 값은 *연산 전에*(`fill_value`) 채워야 원래 값이 산다.
- **교훈**: Series 산술은 **위치가 아니라 라벨로 정렬**한다. 한쪽에만 있는 라벨은 통과가 아니라 `NaN`(전염 + float 승격). 살리려면 `.add(other, fill_value=0)` — `+`나 사후 `.fillna(0)`이 아니라 **연산 전에** 중립값을 채운다.

## 직접 구현 (합격 후 · 검증 → 생산)

### 직접 구현 — 세 채널 판매량 합산(미매칭 보존) 백지에서 짜기
**스펙**: 제품별 판매량을 `online`·`offline`·`app` **세 채널**에서 받음(취급 제품이 조금씩 다름). **제품별 총 판매량(세 채널 합)**을 구하되 **한두 채널에만 있는 제품도 누락 없이** 반영, 결과는 **판매량 내림차순**. 함정 경고 없이 백지 작성.

| 제품 | online·offline·app | 정답 total |
|------|------|:---:|
| 마우스 | 50·40·0 | 90 |
| 스피커 | 0·25·35 | 60 |
| 키보드 | 30·0·10 | 40 |
| 모니터 | 20·15·0 | 35 |
| 웹캠 | 0·0·12 | 12 |

**내 구현 변천 (원문 그대로)**:

**① 1차 — ❌ 에러는 없지만 app이 조용히 증발 (스펙 위반 3곳)**
```python
total = online.add(offline, app, fill_value=0)
print(total)
```
결과 = `online + offline`과 **완전히 동일** → app 통째 무시. 원인: `Series.add(other, level, fill_value, axis)`에서 **두 번째 위치 인자는 `fill_value`가 아니라 `level`** → `app`이 `level` 자리로 들어가 무시됨(**에러조차 안 남 → 더 위험**). 게다가 `.add()`는 상대 Series를 **하나만** 받는데 셋을 한 번에 넣으려 함. 정렬도 누락.
```
마우스 90 / 모니터 35 / 스피커 25 / 키보드 30   (웹캠 없음, 스피커·키보드 틀림)
```

**② 2차 — ✅ 합격**
```python
total = online.add(offline, fill_value=0).add(app, fill_value=0).sort_values(ascending=False)
print(total)
```
**실행 결과** (python 실제 실행)
```
마우스    90.0
스피커    60.0
키보드    40.0
모니터    35.0
웹캠     12.0
dtype: float64
```
→ 스펙 일치. `.add()`는 둘씩만 더하므로 **체이닝**으로 세 번째 채널을 이어 붙이고, 각 단계 `fill_value=0`으로 미매칭을 보존, 끝에 `.sort_values(ascending=False)`.

**교훈**
- `.add()`는 **자기 + 상대 하나**만 더한다. 셋 이상은 **체이닝**(`a.add(b,...).add(c,...)`). 세 번째를 위치로 넣으면 `fill_value`가 아니라 `level` 자리라 **에러 없이 무시**된다.
- **키워드 인자는 이름으로 넘긴다**(`fill_value=`). 위치로 밀어넣으면 엉뚱한 파라미터(`level`)에 꽂혀 조용히 틀린다.
- **에러가 안 나는 게 안전하다는 뜻이 아니다** — 결과를 스펙과 직접 대조해야 "그럴듯한데 틀린" 출력을 잡는다. (다른 소스로 `online+offline`을 재현해 app 누락을 증명.)

## 그날의 핵심
- **Series 산술은 위치가 아니라 라벨(인덱스)로 정렬한다.** 순서가 달라도 같은 라벨끼리 더해진다(정확). 단 **한쪽에만 있는 라벨은 통과가 아니라 `NaN`**(전염 + float 승격) → 값이 조용히 증발.
- **미매칭을 살리려면 `.add(other, fill_value=0)` — 연산 *전에* 채운다.** `+`는 못 하고, 사후 `.fillna(0)`은 이미 사라진 값을 0으로 덮어 **더 틀린다**(값 보존 실패).
- **`.add()`는 둘씩만.** 셋 이상은 체이닝. 세 번째 인자를 위치로 넣으면 `level`로 먹혀 **에러 없이 무시**됨 → 키워드로 명시.
- **(검증 습관)** 에러 없이 돈 코드도 스펙과 대조한다 — 이번 최대 함정은 크래시가 아니라 "조용히 채널 하나가 빠진" 그럴듯한 출력이었다.
