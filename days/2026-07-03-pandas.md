# 📅 2026-07-03 · Day 5 — pandas

## 오늘의 문제

### 오답 — `groupby`는 NaN 그룹 키를 조용히 버린다 (≠ SQL JOIN)
- **문제 (검토한 코드)**:
  ```python
  import pandas as pd

  df = pd.DataFrame({
      "team":   ["A", "A", None, "B", None],
      "points": [10, 20, 30, 40, 50],
  })

  result = df.groupby("team")["points"].sum()
  total  = result.sum()
  print(result); print(total)
  ```
  실제 전체 합 `df["points"].sum()` = 150.
- **내 예측 (그대로)**:
  > "1. result는 각 팀별의 포인트 합을 출력. 코드로만 봤을 땐 70점, 그룹은 A팀 B팀 2개. 2. total은 70점, 전체 point 더한 값과 달라. 3. 그 이유는 기본적인 **inner join 형태**라서 None 값은 포함되지 않고 계산이 돼. 교정: `df.groupby('team')['points', how=left].sum()`"
  > (재시도) "dropna= 를 어디에 붙여야 하는지 모르겠어. result는 A=30, B=40, NaN=80, total=150이 출력돼."

  → **결론 ✅**(None 빠짐·total 70≠150·그룹 2개) / **이유 ❌**(join 아님) / **교정 위치 ❌**(뒤에 정리)
- **실제**:
  ```
  team
  A    30
  B    40      ← result는 값 하나(70)가 아니라 Series. None 팀 행(30·50)이 통째로 빠짐
  total: 70   vs   df['points'].sum(): 150   (80 증발)
  ```
  `groupby`는 두 테이블을 합치는 **join이 아니다** — 한 DataFrame을 키로 **묶는** 것. **기본값 `dropna=True`라 그룹 키가 NaN/None인 행을 조용히 버린다.** join의 inner/outer와 무관.
- **교정 (고친 코드)**:
  ```python
  result = df.groupby("team", dropna=False)["points"].sum()
  #                    ^^^^^^^^^^^^^^^^^^^  groupby()의 인자 (컬럼 선택 ["points"]가 아님)
  # → A=30, B=40, NaN=80, total=150
  ```
  교정 변천: ① `['points', how=left]` → ❌ groupby엔 `how` 없음(merge 인자) + `[]`에 키워드 인자는 문법 에러 · ② `['points', dropna=False]` → ❌ 위치 오류(dropna는 컬럼 선택이 아니라 groupby 옵션) · ③ `groupby("team", dropna=False)` → ✅.
- **교훈**:
  - **pandas `groupby` ≠ SQL `JOIN`.** SQL의 JOIN 드롭 기억을 groupby에 이식하지 말 것.
  - `groupby`는 **기본으로 NaN 그룹 키를 버린다(`dropna=True`)** → 그룹 합의 총계가 원본 총계와 어긋날 수 있다. 살리려면 `dropna=False`.
  - 옵션은 그 동작의 **주체 함수 괄호 안**에. `df[...]`(컬럼 선택)엔 키워드 인자를 못 넣는다(문법 에러).

## 그날의 핵심
- **`groupby`는 join이 아니다.** 한 DataFrame을 키로 묶는 것 — NaN 키 행은 기본(`dropna=True`)으로 조용히 사라진다.
- 그래서 `groupby(...).sum()`의 총합이 원본 총합과 다를 수 있다 → `dropna=False`로 NaN 그룹 살리기.
- 한 도구에서 배운 함정(SQL JOIN 드롭)을 다른 도구(pandas)에 그대로 옮기면 틀린다 — 도구마다 **기본 동작**을 확인.
