# 🧠 OS 이론 T2 — 스케줄링 (2026-07-09)

**개념 씨앗**:
- **프로세스 3상태**: **ready**(CPU만 받으면 실행 가능, 큐 대기) · **running**(CPU 잡고 실행 중, CPU 1개면 한 순간 하나) · **blocked**(I/O 등 대기 중, 지금은 CPU 불필요). 전이: `ready→run`(dispatch) · `run→ready`(퀀텀 만료/선점) · `run→blocked`(I/O 요청) · `blocked→ready`(I/O 완료). **핵심: `blocked`→곧바로 `running` 불가, 반드시 `ready` 경유.**
- **스케줄러** = ready인 것들 중 누구에게 CPU를 줄지 결정.
  - **FCFS**(선착순, 비선점): 도착 순서대로. → **호위 효과(convoy)**: 긴 게 앞이면 뒤가 다 갇힘.
  - **SJF**(최단작업우선, 비선점): 실행시간 짧은 것 먼저 → 평균 대기시간 최소(이론적 최적).
  - **RR**(라운드로빈, 선점): 타임 퀀텀만큼씩 돌아가며 번갈아.
- **선점(preemptive)**: 실행 중인 걸 중간에 뺏어 다른 것에 CPU. **비선점**: 끝나거나 스스로 block할 때까지 안 뺏음.

## 오개념 검증 (4개 판정 → **판정 4/4 정확 · 이유는 B·C 교정**)
| # | 설명 | 내 판정 | 정답 | 판정 |
|:-:|---|:-:|:-:|:-:|
| A | FCFS에서 긴 게 먼저 도착하면 뒤 짧은 것들이 다 기다림 | ⭕ | ⭕ | ✓ |
| B | SJF는 평균대기 최소 최적이니 현대 OS 대부분 SJF를 기본으로 씀 | ❌ | ❌ | ✓ |
| C | RR 퀀텀을 작게 할수록 반응 빨라지니 작을수록 무조건 좋다 | ❌ | ❌ | ✓ |
| D | blocked는 I/O 끝나도 곧바로 running 아니라 ready 거쳐야 running | ⭕ | ⭕ | ✓ |

**판정은 전부 맞음. 그러나 B·D는 이유를 짚어야 함(결론 맞고 근거 어긋남 = T0·T1 패턴 재발).**

- **A ⭕**: 이름 = **호위 효과(convoy effect)**. 긴 트럭 뒤 승용차들이 갇히듯 짧은 작업이 긴 작업 뒤에 묶여 같이 느려짐.
- **D ⭕ (이유 보강)**: 내 이유 "스케줄러의 약속"보다 정확히는 — I/O 끝난 순간 **CPU가 이미 다른 프로세스를 실행 중일 수 있어** 준비됐다고 뺏을 수 없음 → ready에서 순서 대기. CPU가 공유 자원이라 그렇다.
- **B ⭕판정 / ❌이유 (핵심)**:
  - **내 이유 (그대로)**: "평균 대기시간은 줄이게 됐으나, 여전히 순서까지 고려해야 한다는 번거롭고 복잡한 단점이 존재."
  - **실제**: SJF 규칙 자체는 오히려 단순("짧은 거 먼저"). 실무 미채택 진짜 이유 2가지 — ① **실행시간을 실행 전에 알 수 없다**(가장 결정적, 미래를 알아야 성립 → 추측만 가능) ② **기아(starvation)**: 짧은 작업이 계속 오면 긴 작업이 영원히 밀림.
  - **교훈**: "복잡해서"는 **그럴듯한 딴 이유를 갖다 붙인 것**(T1 패턴). 정답 근거 = "**실행시간을 미리 못 안다**." **이론 최적 ≠ 실무 채택.**
- **C ⭕판정 / 이유 미흡**:
  - **내 이유 (그대로)**: "짧은 시간동안 모두 처리를 못하면 다시 뒷차례로 밀려나니, 비효율적이다."
  - **실제**: 밀려나는 것 자체는 손해가 아님(RR은 원래 돌아가며 함). 진짜 손해 = **컨텍스트 스위칭 오버헤드** — CPU를 넘길 때마다 상태(레지스터·PC) 저장/복원 비용 발생. 퀀텀이 너무 작으면 전환이 너무 잦아 일보다 전환에 시간을 더 씀.
  - **수치 감각**: 전환 1회 0.1, 작업 10일 때 — 퀀텀 5→전환 2회(0.2) vs 퀀텀 0.1→전환 100회(10, 일만큼 낭비). **너무 크면 FCFS처럼 느리고, 너무 작으면 전환 낭비 → 트레이드오프.**

## 직접 (트레이스 — SJF가 왜 평균 대기가 짧은지 손 계산)
**스펙**: P1·P2·P3 모두 시각 0 동시 도착, 실행시간 P1=6·P2=2·P3=4. **대기시간** = CPU 처음 받기까지 기다린 시간(= 앞서 실행된 것들의 실행시간 합). FCFS와 SJF의 평균 대기시간을 구하라.

**내 계산 (원문, P3→P1 오타는 스스로 교정)**:
```
FCFS (P1→P2→P3):  P1=0, P2=6, P3=8   →  평균 4.67
SJF  (P2→P3→P1):  P2=0, P3=2, P1=6   →  평균 2.67
```
**검증**: FCFS = (0+6+8)/3 = 14/3 = 4.67 ✅. SJF는 짧은 순 P2(2)→P3(4)→P1(6) = (0+2+6)/3 = 8/3 = 2.67 ✅. **둘 다 정확.**

**해석**: FCFS는 긴 P1(6)을 앞에 둬 뒤 둘이 6을 통째로 대기. SJF는 짧은 P2(2)를 앞에 둬 뒤가 지는 지연이 작음 → 평균 거의 절반. **"짧은 걸 먼저 = 뒤에 줄 선 모두의 대기 총량↓"** 이 SJF 최적성의 정체. 단 이 표의 실행시간(6·2·4)을 **현실 OS는 실행 전에 모른다**(문제에선 줬을 뿐) → B의 "이론 최적이나 실무 불가"가 여기서 완성.

## 그날의 핵심
- **상태 전이**: `blocked`는 I/O가 끝나도 곧바로 `running` 불가 — 반드시 `ready` 경유(CPU가 남에게 점유됐을 수 있어 스케줄러가 다시 뽑아야 함).
- **SJF는 평균 대기시간 이론적 최적**(짧은 걸 앞에 둬 뒤의 대기 총량을 줄임 — 4.67 vs 2.67로 직접 확인). **하지만 실무 미채택: 실행시간을 미리 못 앎 + 기아.** 이론 최적 ≠ 실무 채택.
- **RR 퀀텀은 트레이드오프**: 너무 크면 FCFS처럼 반응 느리고, 너무 작으면 **컨텍스트 스위칭 오버헤드**로 낭비. "작을수록 무조건 좋다"는 비용 무시.
- **🔴 결론 맞고 이유 어긋남 재발**(B·C): T0(권한=속도)·T1(비용=속도)과 같은 패턴. 판정이 맞아도 **근거가 정답인지** 매번 되묻기 — "복잡해서/밀려서" 같은 그럴듯한 딴 이유를 경계.
