# 🧠 OS 이론 T0 — OS란·구조 (2026-07-03)

**개념 씨앗**: OS = 하드웨어(CPU·메모리·디스크·네트워크)를 관리하고 프로그램에 나눠주는 **중간 관리자**(프로그램은 하드웨어 직접 못 만짐). **커널** = 하드웨어를 직접 다루는 특권부. **유저 모드**(일반 프로그램·제한된 권한) vs **커널 모드**(커널·전권) — 나눈 이유는 **보호**. **시스템 콜** = 유저 프로그램이 커널에 기능을 요청하는 창구(이때 user→kernel 모드 전환).

## 오개념 검증 (4개 판정)
| # | 설명 | 내 판정 | 정답 | |
|:-:|---|:-:|:-:|:-:|
| A | 일반 프로그램이 하드웨어를 **직접 제어**한다 | ⭕ | **❌** | ✗ |
| B | `print`도 내부적으로 시스템 콜을 거친다 | ⭕ | ⭕ | ✓ |
| C | 커널 모드는 "**더 빠른**" 실행 모드다 | ⭕ | **❌** | ✗ |
| D | 시스템 콜은 user→kernel 모드 전환을 일으킨다 | ⭕ | ⭕ | ✓ |

- **A (정답 ❌)**: 프로그램은 하드웨어를 직접 제어 못 한다 — 커널에 `write()` 등 **시스템 콜로 요청**, 커널이 하드웨어를 다룬다. 유저/커널 모드를 나눈 이유 자체가 "앱이 하드웨어를 직접 못 건드리게".
- **C (정답 ❌)**: 커널 모드 = **권한(전권)** 차이지 **속도**가 아니다. 내 직감 "명령체계를 직접 다룬다"는 방향(=권한)은 맞았으나 "더 빠르다"는 잘못된 결론. 커널 모드라고 코드가 빨리 도는 게 아니다.
- **핵심 패턴**: 유저 vs 커널 = **"권한(직접 접근 O/X)" 차이지 "속도" 차이가 아니다.** 프로그램은 하드웨어를 항상 커널을 거쳐 만진다.

## 직접 (Feynman) — 메모장 `Ctrl+S` 저장
- **내 설명 (그대로)**: "메모장은 유저모드로 실행 → Ctrl+S로 저장 요청을 커널에 보냄 → 커널모드 작동 → 저장 진행 = 시스템 콜." → ✅ **합격** (핵심 체인 정확). 특히 "직접 저장"이 아니라 "커널에 요청"이라 한 것 = **A 오개념 스스로 교정**.
- **정답 흐름 (보강)**:
  1. 메모장 = 유저 모드 실행 중
  2. Ctrl+S → 메모장은 디스크 직접 못 만짐 → 커널에 "파일에 써줘" = `write()` 시스템 콜
  3. 시스템 콜 → CPU가 유저 → 커널 모드 전환
  4. 커널(전권)이 실제로 디스크(하드웨어)에 기록
  5. 끝나면 유저 모드로 복귀 → 메모장 계속 실행

## 그날의 핵심
- OS/커널은 하드웨어를 관리하고, 프로그램은 **커널을 거쳐야만** 하드웨어를 쓴다(직접 ❌).
- 유저 vs 커널 = **권한** 차이(전권/제한)지 **속도** 차이가 아니다.
- **시스템 콜** = 유저 프로그램이 커널에 요청하는 창구 + user→kernel 모드 전환의 관문. (`echo`/`print`도 내부적으로 `write()` 시스템 콜)
