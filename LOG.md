# 검증 드릴 오답 로그

형식: 날짜 / 영역 / 내가 한 예측 / 실제 정답 / 핵심 교훈

## 2026-06-29 — SQL (검증형: 유저별 총 결제액 쿼리 리뷰)

### 오답 1 — JOIN 행 복제 ≠ WHERE 필터
- **내 예측**: `status = 'done'` 조건을 걸었기 때문에 금액이 실제보다 크게 나올 수 없다.
- **실제**: 크게 나올 수 있다. 금액 뻥튀기의 원인은 주문 필터(WHERE)가 아니라 **JOIN이 행을 복제**하는 것이다. status 필터는 어떤 주문을 포함할지만 거를 뿐, 행 복제를 막지 못한다.
- **교훈**: JOIN을 거칠 때마다 "이 JOIN이 행을 늘리나?"를 먼저 묻는다. 결제를 주문 단위로 먼저 합치거나(서브쿼리/사전집계) JOIN 구조로 해결해야지, WHERE로는 못 막는다.

### 오답 2 — 집계 GROUP BY는 식별자(id)로
- **내 예측**: SELECT 컬럼이 2개라서 GROUP BY에 두 컬럼 모두 써야 한다.
- **실제**: SUM은 집계라 GROUP BY에 안 넣어도 되어 문법은 통과한다. 진짜 위험은 **name이 유저를 유일하게 식별하지 못한다**는 것. 동명이인이 있으면 서로 다른 사람의 결제액이 한 줄로 합쳐진다.
- **교훈**: 사람을 식별하는 건 이름이 아니라 id. `GROUP BY u.id`(보여줄 땐 `GROUP BY u.id, u.name`)로 묶어야 한다.

### 잘한 것
- INNER JOIN이라 결제 없는 'done' 주문은 결과에서 빠진다는 것을 정확히 예측함. (모두 살리려면 LEFT JOIN → amount NULL)

### 다음 개선 포인트
- 코드를 위→아래 직관으로만 읽지 말고, **데이터를 머릿속에서 한 행씩 굴려 반례를 떠올리기**. "order 10에 payment 2건이면? 동명이인이면?"

### 🔁 복기 (마스터리 루프)
같은 개념을 *다른 유형*으로 재출제해 합격까지 반복. (표면만 바꾸고 핵심 함정은 동일)

- **오답 2 (집계는 식별자로 GROUP BY)** — **1회 합격**.
  - 재출제: "브랜드별 총 재고"(`GROUP BY b.name`). 동명 브랜드가 한 줄로 합쳐지는 위험을 정확히 짚고 `GROUP BY b.id`로 교정. ✅
- **오답 1 (JOIN 행 복제 ≠ WHERE 필터)** — **3회 만에 합격**.
  - 1차(상품×프로모션×판매 SUM): 또 "WHERE가 막아준다"고 오판 → 불합격. *오늘 오답 1과 같은 함정 반복.*
  - 2차(고객×주문×쿠폰 SUM): **"WHERE는 행 복제와 무관"을 드디어 맞힘**(핵심 진전). 그러나 JOIN을 직접 펼치지 않아 행 수·합계 오답 → 불합격.
  - 3차(같은 데이터를 GROUP BY/SUM 떼고 6행=주문2×쿠폰3 전부 전개): total=9000=실제 3000의 **3배(=쿠폰 수)**, "각 주문×각 쿠폰 경우의 수로 부풀고 WHERE는 복제를 못 되돌린다"고 정확히 설명 → ✅ 합격.

### 강화된 교훈
- 집계 쿼리는 **SUM/COUNT 전에 "이 JOIN이 행을 늘리나?"를 먼저 묻는다.**
- 의심되면 **GROUP BY/SUM을 떼고 JOIN 결과를 한 행씩 펼쳐서** 직접 센다. (머릿속 직관 < 실제 행 전개)
- `WHERE`는 *어떤 행을 통과시킬지*만 거른다. **이미 복제된 행을 다시 합쳐 줄이지 못한다.**
