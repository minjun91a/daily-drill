# 매일 코딩 검증 드릴 (daily-drill)

AI가 짠 코드를 **읽고 / 흐름을 따라가고 / 틀린 곳을 짚고 / 어떻게 고치는지까지** 설명하는 능력을 기르는 하루 한 문제 드릴. 타이핑·문법 암기가 아니라 **검증·디버깅하는 눈**을 키우는 게 목표다.

## 왜 하는가
구현을 AI에 위임해오면서 라인 단위 코드를 "읽고 검증"하는 근육이 약해졌다. 이 드릴은 그 갭을 메운다. 동시에 AI가 **"자신 있게 틀리는" 패턴**을 모아, 멀티에이전트 오케스트레이션 엔진 `mint-agent`의 trust layer 검증 규칙 후보이자 포트폴리오 자산으로 쌓는다.

2. **① 진단(예측 우선)** — *돌리기 전에* 결과·문제점을 글로 먼저. (가장 중요, 생략 금지)

## 📚 누적 교훈 (전 기간)
- 집계 쿼리는 `SUM`/`COUNT` 전에 **"이 JOIN이 행을 늘리나?"** 를 먼저 묻는다.
- 의심되면 `GROUP BY`/`SUM`을 떼고 **JOIN 결과를 한 행씩 펼쳐서** 직접 센다.
- `WHERE`는 통과 여부만 거른다. **이미 복제된 행을 다시 합쳐 줄이지 못한다.**
- **교정도 검증한다**(진단 → 교정 → 교정 검증): `SUM(DISTINCT)` 같은 함정 fix 회피.
- 묶는 기준은 라벨(name)이 아니라 **식별자(id)**.
- **(pandas)** `merge` = `JOIN`. 오른쪽 키 중복 → fan-out. `how`는 복제 제어 못 함. 안 쓰는 테이블은 merge 안 함(`validate="m:1"` 가드).
- **(Python)** mutable 기본 인자(`def f(x, items=[])`)는 정의 시 1번 생성 → 호출 간 공유. 새 객체는 몸통 안에서(`items=None` + `is None`).
- **(Python)** `None` 비교는 항상 `is None`(`== None` 아님) — 싱글턴 정체성 / `__eq__` 속임 / 배열은 `ValueError`.

## 다음 개선 포인트
- 코드를 직관으로만 읽지 말고 **데이터를 한 행씩 굴려 반례** 떠올리기.
- **다음 SQL 날**: "JOIN 행 복제"를 스캐폴드 없이 **cold**로 재확인 → 혼자 진단+교정까지 가면 마스터.
- **pandas** merge fan-out: cold 1차에 'how=left'로 회귀 → "오른쪽 키 유니크?" 자동 점검 습관 + `validate=` 가드 손에 익히기.
