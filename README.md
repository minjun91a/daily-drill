**AI가 짠 코드를 그대로 믿지 않고 → 돌리기 전에 결과를 예측하고 → 틀린 곳을 짚고 → 직접 고치는** 눈을 기르는 매일 연습. 타이핑·문법 암기가 아니라 "읽고 검증하고 디버깅하는" 실무 능력이 목표.

## 이 훈련의 방식 (여기가 핵심)
1. **문제** — "AI/동료가 짠 코드인데 어딘가 틀렸다" 형태의 코드를 받는다.
2. **예측 먼저** — 코드를 *돌리기 전에* "이 코드는 이런 결과가 날 것이다, 왜냐하면…"을 **글로 먼저 적는다.** (가장 중요)
3. **대조** — 실제 실행 결과와 내 예측을 맞춘다. **어긋난 지점 = 내가 잘못 알고 있던 것.**
4. **고치기** — 원인을 짚고 **올바른 코드를 직접 작성**한다. (예측 + 고치기까지 맞아야 통과)
5. **틀리면 다시** — 같은 개념을 *다른 유형*의 문제로 다시 내서, 맞출 때까지 반복한다.

> **차별점**: 내가 *틀리게 예측한 문장을 원문 그대로* 남긴다. 나중에 "그때 나는 이렇게 잘못 생각했고, 이렇게 고쳤다"가 코드로 다 보인다 — 이 기록은 학습 노트가 아니라 **내 사고를 검증·교정한 증거**다.

## 💬 이렇게 훈련합니다 (대화 예시)

> 이 훈련이 실제로 어떻게 굴러가는지 — 아래는 축약본. 전체는 [`days/`](days/)에.

**문제** — 유저별 총 결제액(완료 주문만). 이 합계, 실제보다 커질 수 있을까?
```sql
SELECT u.name, SUM(o.amount) AS total
FROM users u
JOIN orders   o ON o.user_id  = u.id
JOIN payments p ON p.order_id = o.id     -- 주문 1 : 결제 N
WHERE o.status = 'done'
GROUP BY u.name;
```

**내 예측 (틀림, 원문 보존)** — "`status = 'done'` 조건을 걸었기 때문에 금액이 실제보다 크게 나올 수 없다."

**실제** — 커질 수 있다. 원인은 WHERE가 아니라 **JOIN이 행을 복제**하는 것. 한 주문에 결제가 2건이면 `o.amount`가 두 번 더해진다. status 필터는 *어떤 주문을 포함할지*만 거를 뿐 복제를 못 막는다.

**교정** — '결제 있음'은 JOIN이 아니라 `EXISTS`로 걸어 복제를 원천 차단:
```sql
SELECT u.id, u.name, SUM(o.amount) AS total
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE o.status = 'done'
  AND EXISTS (SELECT 1 FROM payments p WHERE p.order_id = o.id)
GROUP BY u.id, u.name;
```

**교훈** — 집계 전 "이 JOIN이 행을 늘리나?"를 먼저 묻는다. WHERE는 행을 거를 뿐, 복제는 못 막는다.

## 🔍 AI가 자주 틀리는 패턴 — 검증 체크리스트
> 훈련에서 반복해 마주친 "그럴듯한데 틀린" 패턴들. 내 멀티에이전트 엔진 **mint-agent**의 검증 계층(trust layer)에 넣을 **자동 검증 규칙 후보**로 쌓는 중이다.

- **집계 전에 "이 JOIN/merge가 행을 늘리나?"를 먼저 묻는다.** 늘어난 행이 `SUM`을 부풀린다.
- **`WHERE`·`how` 같은 필터/조인 방식은 행 복제를 막지 못한다.** (필터는 "어떤 행을 통과시킬지"만 정할 뿐)
- **집계는 이름이 아니라 식별자(id)로 묶는다.** 동명이인·동명 브랜드가 한 줄로 합쳐지는 사고 방지.
- **고친 코드도 다시 검증한다.** 손쉬운 수정(`SUM(DISTINCT)` 등)이 또 다른 버그일 수 있다.
- **`GROUP BY`는 JOIN으로 늘어난 행을 다시 접는다** — 출력 행 수 = 그룹 종류 수. 집계값은 전체 합 하나가 아니라 **그룹마다 하나**.
- **LEFT JOIN 빈 그룹: `COUNT`은 0, `SUM`은 NULL**(0 아님). 0으로 보이려면 `COALESCE(SUM(x), 0)`. 그리고 빈 쪽을 0으로 세려면 `COUNT(*)`(행을 셈, 1) 말고 `COUNT(조인테이블.컬럼)`(NULL 아닌 값, 0).
- **`!=`·`<>`·`IN`·`NOT IN`은 NULL을 조용히 버린다.** NULL 비교는 TRUE/FALSE가 아니라 **UNKNOWN**이고 `WHERE`는 결과가 TRUE인 행만 통과시킨다 → NULL 행이 소리 없이 사라진다. `!= 'x'`를 쓸 때마다 "이 컬럼에 NULL 있으면?"을 묻고, 살리려면 `컬럼 IS NULL`을 `OR`로 명시. (모든 조건은 자기 컬럼을 다시 쓴다 — `IS NULL`도 왼쪽에 컬럼 필요.)
- **"A도 아니고 B도 아니다" = `!= A AND != B` (= `NOT IN (A, B)`).** `OR`로 이으면 값이 A·B 중 하나만 되므로 **항상 참** → 아무도 못 거른다. 부정을 나눠 넣을 땐 연결도 뒤집는다(De Morgan: `NOT(A OR B) = NOT A AND NOT B`). `AND`는 `OR`보다 우선순위가 높다 — 섞으면 괄호로 의도를 못박는다.
- **(pandas)** `merge`는 SQL `JOIN`과 같다. 붙이는 쪽 키가 중복이면 행 복제(fan-out). 안 쓰는 프레임은 붙이지 않고, 붙일 땐 `validate="m:1"`로 사전 검증.
- **(pandas)** `groupby`는 SQL `JOIN`과 **다르다** — 한 DataFrame을 키로 묶는 것. 기본 `dropna=True`라 **NaN 그룹 키 행을 조용히 버려** 총합이 원본과 어긋난다. 살리려면 `dropna=False`.
- **(pandas)** **Series 산술(`+`·`add`)은 위치가 아니라 라벨(인덱스)로 정렬**한다. 순서가 달라도 같은 라벨끼리 더해지지만, **한쪽에만 있는 라벨은 통과가 아니라 `NaN`**(어떤 숫자와 더해도 `NaN`, dtype은 float로 승격) → 값이 조용히 증발. 살리려면 `.add(other, fill_value=0)`으로 **연산 전에** 채운다(사후 `.fillna(0)`은 이미 사라진 값을 0으로 덮어 **더 틀린다**).
- **(pandas)** `.add()`는 **둘씩만** 더한다(자기+상대 하나). 셋 이상은 **체이닝**(`a.add(b,...).add(c,...)`). 키워드는 이름으로 넘긴다 — 세 번째를 위치로 넣으면 `fill_value`가 아니라 `level` 자리라 **에러 없이 무시**된다. **에러가 안 나도 스펙과 대조**해야 "조용히 한 소스가 빠진" 그럴듯한 출력을 잡는다.
- **(pandas)** **chained indexing으로 대입하면(`df[mask][col] = 값`) 원본이 안 바뀐다.** `df[mask]`가 **복사본**을 만들고 대입은 그 복사본에 쓰인 뒤 버려진다(pandas 3.0은 `ChainedAssignmentError`로 막음). 대입문을 볼 때 "원본을 바꾸나, 복사본을 바꾸나?"를 먼저 묻는다. 부분 행 수정은 **`df.loc[행 마스크, '열 라벨'] = 값`** — `df`를 한 번에 인덱싱해 원본에 직접 쓴다. **`.loc[행, 열]`의 두 자리는 성격이 다르다**: 행=값으로 만든 조건(`df['x'] < n`), 열=이름 라벨(`'col'`, `df['col']` 아님). 한 자리 문법을 다른 자리에 이식하면 엉뚱한 컬럼을 **에러 없이** 만든다. 마스크는 `==`(비교)이지 `=`(대입)이 아니다.
- **(Python)** 함수 기본값에 `[]`·`{}`를 두면 정의될 때 한 번만 만들어져 호출 간 공유된다. 매번 새 값이 필요하면 함수 안에서 만든다.
- **(Python)** `None` 비교는 항상 `is None`으로 한다. (`== None`은 커스텀 객체·배열에서 틀린 결과나 에러)
- **(Python)** 리스트를 **순회하며 그 리스트를 수정**(`remove`/`del`)하면 index가 밀려 원소를 건너뛴다. 사본 순회(`nums[:]`) 또는 컴프리헨션으로 새 리스트를 만든다.
- **(Python)** **중첩 리스트를 `*`로 만들면 안쪽을 참조 공유**한다 — `[[0]*3]*3`은 같은 행 하나를 3번 가리켜, 한 칸 수정이 전 행에 번진다. 독립된 행은 컴프리헨션 `[[0]*3 for _ in range(3)]`. 근본: **`*`·함수 기본값·대입(`b=a`)은 복사가 아니라 참조 증가** — "따로인 줄 알았는데 하나". 진짜 복사는 컴프리헨션·`copy`·`[:]`.

## 왜 이 훈련을 하는가
AI가 코드를 대신 써주는 시대일수록, 그 코드를 **읽고·검증하고·디버깅하는 능력**이 개발자의 진짜 경쟁력이 된다. 이 레포는 그 능력을 매일 의식적으로 훈련한 기록이다. "구현을 AI에 위임할수록 라인 단위 검증 근육이 약해진다"는 문제를 스스로 진단하고, 그 약점을 재능이 아니라 **시스템(매일 반복 · 예측 우선 · 오답 발견)**으로 메운다. 여기서 축적한 검증 패턴은 멀티에이전트 엔진 [mint-agent](https://github.com/minjun91a/mint-agent)의 trust layer 규칙으로 이어진다.

## 진행 방향
- 코드를 위→아래 직관으로 읽지 않고 **데이터를 한 행씩 굴려 반례를 찾는** 검증 절차를 기본기로 정착시킨다.
- 반복해 마주친 함정(JOIN·merge fan-out, mutable 기본 인자 등)을 힌트 없이 맨몸으로 재현·교정할 수 있을 때까지 연습한다.
- 검증 체크리스트를 mint-agent trust layer의 **실제 규칙 코드**로 옮겨 "학습 → 아키텍처"로 자산화한다.
