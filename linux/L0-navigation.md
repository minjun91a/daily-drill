# 🐧 Linux L0 — 탐색·이동 (2026-07-03 · Git Bash)

**개념 씨앗**: `pwd`(현재 경로) · `ls`(목록) · `cd 경로`(이동). 경로 기호 `/`(루트) `~`(홈) `.`(현재) `..`(부모). 절대경로(`/`·`~`로 시작, 어디서든 같은 곳) vs 상대경로(현재 위치 기준).

트리:
```
/home/mint/
├── projects/  ├── web/   └── data/
└── docs/
```

## 예측 — cd 연쇄 (시작: `/home/mint/projects/web`)
| # | 명령(이어서) | 내 예측 | 실제 | |
|:-:|---|---|---|:-:|
| 1 | `cd ..` | `/home/mint/projects` | 같음 | ✅ |
| 2 | `cd data` | `/home/mint/projects/data` | 같음 | ✅ |
| 3 | `cd ../../docs` | `/home/mint/docs` | 같음 | ✅ |
| 4 | `cd ~` | **`/home`** | `/home/mint` | ❌ |

**오답 4 — `~` vs `/home`**: `~` = 내 홈 폴더(`/home/mint`) 하나. `/home`은 그 홈들을 담는 **부모**. `cd ~`는 내 집으로 가지, 한 칸 위 동네(`/home`)로 가는 게 아님.
- 증거: `echo ~` = `/c/Users/최민준` — `~`는 늘 "특정 사용자 전용 폴더"의 전체 경로.

## 개념 오해 교정 — `~`는 최상위가 아니다
> 내 질문: "cd ~ 는 폴더트리 최상위로 가는 거 아냐?" → **❌**
- **최상위 = `/`(루트).** `cd /` → `pwd` = `/`.
- `~` = `/home/mint` = 루트에서 **두 칸 아래**(`/` → `home` → `mint`)의 내 홈.
```
/              ← 최상위 (cd /)
└── home/
    └── mint/  ← ~ (cd ~) · 꼭대기 아님, 두 칸 아래
```
- 터미널이 홈에서 시작해 "기준점"처럼 느껴질 뿐. 꼭대기로는 `cd /`.

## 직접 — `/home/mint/docs`에서 `web`으로 (3방법)
| 방법 | 내가 쓴 명령 | 판정 |
|---|---|---|
| (a) 상대 | `cd projects/web` | ❌ docs 안엔 projects 없음(형제) → **`cd ../projects/web`** |
| (b) 절대 | `cd /home/mint/projects/web` | ✅ |
| (c) `~` | `cd ~/projects/web` | ✅ (`~=/home/mint` 정확 적용) |

실행 확인: `cd projects/web` → '그런 폴더 없음' / `cd ../projects/web` → `/home/mint/projects/web` 도착.

## 그날의 핵심
- 상대경로는 **지금 위치 기준**. 이동 전 "지금 어디? 목표가 자식/형제/부모?"를 먼저 판단.
- 목표가 **형제**면 `..`로 부모까지 올라갔다 내려온다(`../형제/...`).
- 세 개는 전부 다른 곳: `~`(내 홈=`/home/mint`) ≠ `/home`(홈들의 부모) ≠ `/`(최상위 루트).

# 🔁 L0 2회차 (2026-07-24 · WSL Ubuntu 24.04)

> L6(텍스트 처리)를 시작하려다 **"아직 기초가 확립되지 않았다"**고 스스로 판단해 L0부터 다시. 환경도 Git Bash → **진짜 리눅스(WSL)**로 바꿔, 홈이 `/home/min`인 실제 경로 감각으로 재훈련. 연습 트리 `~/drill-l0/{projects/{web/src,api},docs/notes,tmp}`.

## ② 세 랜드마크 — 1회차 오답 재시험 ✅ 전부 정답

| # | 명령 | 내 예측 (그대로) | 실제 | |
|:-:|---|---|---|:-:|
| A | `cd ~` → `pwd` | `/home/min` | `/home/min` | ✅ |
| B | `cd /` → `pwd` | `/` | `/` | ✅ |
| C | `cd /home` → `pwd` | `/home` | `/home` | ✅ |

**1회차에 `cd ~`를 `/home`으로 예측하고 "`~`가 최상위 아니냐"고 오해했던 지점을 이번엔 셋 다 정확히 구분.** 추가 확인: `/`에서 `cd ..` → 여전히 `/`(꼭대기의 부모는 자기 자신) · `~`에서 `cd ..` → `/home` · `ls /home` → `min` 하나.

## ③ 적용 예측

| # | 문제 | 내 예측 (그대로) | 실제 | |
|:-:|---|---|---|:-:|
| Q1 | `~/drill-l0/docs`의 전체 경로 | `/home/min` | `/home/min/drill-l0/docs` | ❌ |
| Q2 | web에서 `cd ../api` | `/home/min/drill-**10**/projects/api` | `/home/min/drill-**l0**/projects/api` | 🟡 |
| Q3 | web에서 `cd ~/docs` | 실패 | 실패(`No such file or directory`) | ✅ |

**Q1 오답 — `~`는 "목적지"가 아니라 "앞 조각 치환"**: `~`는 경로의 맨 앞 한 조각만 `/home/min`으로 갈아끼우고 **뒤에 붙은 부분은 그대로 따라온다**. `/home/min`만 답한 건 `~`만 풀고 멈춘 것.
- 단 **Q3에서는 같은 개념을 정확히 적용**(`~/docs` → `/home/min/docs`, docs는 `drill-l0` 안이라 없음)했으므로 개념 부재가 아니라 질문을 좁게 읽은 문제.

**Q2 — 경로 논리 ✅, 이름 ❌**: `../api`로 형제에 간다는 판단은 정확(1회차에 `../`를 빠뜨렸던 지점 극복). 다만 `drill-l0`(엘·영)을 `drill-10`(십)으로 표기.
- 증거: `cd ~/drill-10` → `No such file or directory`
- 헷갈림 조합: `l`(엘)↔`1`(일)↔`I`(아이), `0`(영)↔`O`(오)

## ④ 검증 함정 — 동료의 메모

**문제**: 동료가 *"web으로 오려면 `cd drill-l0/projects/web`"* 라고 남김. 나는 `/home/min/drill-l0/docs`에 서 있다. ⑴ 실행하면? ⑵ 근본 문제는? ⑶ 어디서 실행해도 되게 고치면?

**내 예측 (그대로)** —
> 1. 실행되지 않는다. 그 이유는 drill-l0 앞에 ~/ 를 안붙여서
> 2. 이 위치라면 drill-l0 하위 경로로 동일선상에 있는 docs 폴더와 충돌이 일어나기 때문.
> 3. ~/drill-l0/projects/web 이라고 작성해야 한다. 또는 cd ../projects/web

**실제** — 위치를 바꿔가며 실제 실행한 결과:
```
[1] docs 에서   cd drill-l0/projects/web   ->  실패
[2] 홈  에서    cd drill-l0/projects/web   ->  성공! /home/min/drill-l0/projects/web
[3] docs 에서   cd ../projects/web         ->  성공!
[4] 홈  에서    cd ../projects/web         ->  실패
[5] /tmp 에서   cd ~/drill-l0/projects/web ->  성공!
```
- **⑴ 결과 ✅ / 이유 🟡** — 원인은 "`~/`를 안 붙여서"(고치는 방법)가 아니라 **"상대경로라 셸이 지금 있는 `docs` 안에서 `drill-l0`를 찾았고, 거기엔 `notes`뿐이라 없었다"**(셸이 어디를 뒤졌나). 원인은 *무엇이 빠졌나*가 아니라 *셸의 탐색 위치*로 설명해야 한다.
- **⑵ ❌ "충돌" 오해** — 부딪히는 건 아무것도 없다. 근본 문제는 **상대경로는 실행 위치에 따라 결과가 달라진다**는 것. 그리고 `[2]`가 보여주듯 **동료는 홈에 서서 그 메모를 썼다** → 메모가 틀린 게 아니라 **기준 위치를 안 적은 것**이 문제.
- **⑶ 절반 ✅** — `~/drill-l0/projects/web`은 정답(`[5]` /tmp에서도 도착). 그러나 `cd ../projects/web`은 **상대경로**라 `[4]`처럼 홈에서 실패 → "어디서든"이라는 조건 위반. 두 번째 정답은 완전 절대경로 `/home/min/drill-l0/projects/web`.

**교훈** — 경로를 **남길 때**(스크립트·메모·문서)는 절대경로, 지금 이 자리에서 잠깐 이동할 때만 상대경로. "어디서든 되어야 한다"가 조건이면 무조건 절대경로.

## ⑤ 직접 — 4회 만에 합격

**스펙** — 출발 `/home/min/drill-l0/projects/web/src`. ⑶개를 각각 한 줄로: **(a)** `docs/notes`로 **상대경로만** · **(b)** 같은 목적지로 **`/`로 시작하는 절대경로** · **(c)** **홈으로** 한 번에.

❌ **1차** — 세 개 모두 실패:
```bash
a. cd ../../../home/min/drill-l0/docs/notes
b. cd /docs/notes
c. cd ~/docs/notes
```
- **(a)** `..` 3번은 **정확**(→ `/home/min/drill-l0` 도착, 실행으로 확인). 그런데 **이미 도착한 자리에서 전체 경로를 다시 이어붙임** → 셸은 `/home/min/drill-l0/home/min/...`을 찾다 실패. *이미 건물 안에 들어와 있는데 주소를 시(市)부터 다시 적은 격.*
- **(b)** 절대경로를 "`/`로 시작"으로만 이해. 실제 루트 밑엔 `bin boot dev etc home ...`뿐이고 `docs`는 없음 → **루트에서 목표까지 전 구간**을 적어야 한다.
- **(c)** 목적지 오독(홈 ≠ `docs/notes`) + `~/docs`는 **Q3에서 스스로 "실패한다"고 맞혔던 바로 그 경로**를 재사용.

❌ **2차** — (a) 합격, 나머지 실패:
```bash
a. cd ../../../docs/notes          # ✅ "2칸 = 결국 답은 이거네"
b. cd home/min/drill-l0/docs/notes # 맨 앞 / 누락
c. ~/drill-l0/docs/notes           # 목적지 오독 반복
```
- **(a) ✅** — "`../../../`로 `drill-l0`에 서 있으니 남은 건 `docs/notes` 2칸"으로 스스로 조립. 경로 조립법을 잡은 순간.
- **(b)** 사슬은 한 칸도 안 틀렸고 **맨 앞 `/` 하나**만 빠짐. `home/...`은 "지금 자리에서 출발", `/home/...`은 "루트에서 출발" — **슬래시 하나가 절대↔상대를 가른다.**
- **(c)** **에러 없이 엉뚱한 곳 도착** — 명령은 성공했지만 도착지가 `/home/min/drill-l0/docs/notes`로 홈이 아님. (전날 pandas Day 14의 "에러 없이 그럴듯한 오답"과 같은 패턴)

❌ **3차**:
```bash
b. cd /home/min/dirll-l0/docs/notes   # 철자: dirll → drill
c. cd ~/home/min                      # ~ 를 접두사로 오해
```
- **(b)** 구조는 통과(`/` 붙임 ✅). 철자만 어긋남 → **오늘만 두 번째 이름 실수**(`drill-10`, `dirll-l0`).
- **(c)** 핵심 개념 미확립: `~`는 **그 자체가 이미 `/home/min` 전체**인 완성된 주소지 접두사가 아니다.
  ```
  ~            -> /home/min            (이미 홈. 끝)
  ~/drill-l0   -> /home/min/drill-l0   (~ 뒤에 붙이면 홈 "안으로")
  ~/home/min   -> /home/min/home/min   (존재하지 않음)
  ```
  *`~`는 "우리 집"이라는 말. `~/home/min`은 "우리 집 안의 → home → min"이라고 말한 것.*

✅ **4차 — 합격**:
```bash
b. cd /home/min/drill-l0/docs/notes   # -> /home/min/drill-l0/docs/notes
c. cd ~                               # -> /home/min (= 홈)
```

**교훈** — 경로 조립 2단계: ⑴ 트리에서 **출발점과 목표점을 각각 찍는다** ⑵ **상대**면 `..`로 올라간 뒤 *"지금 여기 서 있다"*고 되뇌고 **남은 길만** 잇고, **절대**면 `/`에서 목표까지 **모든 칸**을 적는다.

## 🔑 새로 익힌 것 — Tab 자동완성

이름 실수가 하루에 두 번 나와서 도입. 경로를 끝까지 타이핑하지 않는다:
```
cd /home/min/dr   ← 여기까지 치고 Tab
cd /home/min/drill-l0/   ← 셸이 자동 완성
```
- **존재하는 이름만 완성된다** → 오타면 아예 완성이 안 되어 **치는 순간 바로 안다.**
- 후보가 여러 개면 `Tab` 두 번 → 목록 표시.
- 실무에서 긴 경로를 손으로 다 치는 사람은 없다. 거의 전부 Tab.

## 2회차-심화 (같은 날 · "한 번 더" · 새 트리 `~/lab`)

외운 트리로 맞히는 걸 막으려 구조를 새로 만듦(`~/lab/{src/core/utils,build,docs/api,logs}`). 출발 `/home/min/lab/src/core/utils`.

**② 순차 예측 5문 — 전부 정답** (각 줄이 이전 위치에서 이어짐):
```
Q1 cd ../..           -> /home/min/lab/src           ✅
Q2 cd ~/lab/docs/api  -> /home/min/lab/docs/api       ✅
Q3 cd ~/lab/logs      -> /home/min/lab/logs           ✅
Q4 cd /lab/build      -> 실패(루트에 lab 없음)          ✅ ← 이유까지 정확
Q5 cd ~               -> /home/min                     ✅
```
- Q1의 `../..`(utils→core→src) 두 칸을 안정적으로 셈. Q4는 *"`/`로 시작하니 루트에서 찾는데 루트 밑에 lab이 없다"*고 스스로 설명 — 2회차 (b)에서 틀렸던 절대경로 개념을 뒤집음.
- 유일한 흠집: pwd 결과를 `home/min/...`처럼 **맨 앞 `/` 없이** 표기. → **`pwd`는 늘 `/`로 시작한다 = 현재 위치는 언제나 절대경로로 표현된다.**

**⑤ 직접 — 한 번에 합격** (출발 `/home/min/lab/docs/api`):
```bash
a. cd ../../src/core/utils/   # -> /home/min/lab/src/core/utils  (api→docs→lab 올라가 재하강)
b. cd /home/min/lab/build/    # -> /home/min/lab/build           (절대경로 완벽)
c. cd ~/lab/logs/             # -> /home/min/lab/logs            (~ 정확)
```
2회차에서 4회 헤매던 직접 구현을 심화 회차에선 **1회 합격**. "아는 것 → 쓸 수 있는 것"으로 전이 확인. (끝의 `/`는 폴더 이동에선 무해.)

## 2회차 핵심
- **`~`는 접두사가 아니라 완성된 주소 그 자체**(`= /home/min`). 홈에 가려면 `~` 뒤에 아무것도 붙이지 않는다. 뒤에 붙이는 건 "홈 *안으로* 들어가는" 것.
- **`pwd`는 항상 `/`로 시작한다** — 현재 위치는 언제나 절대경로. 경로를 적을 때 맨 앞 `/`를 의식한다.
- **맨 앞 `/` 하나가 절대경로와 상대경로를 가른다.** `/home/...`=루트에서 출발, `home/...`=지금 자리에서 출발.
- **`..`로 올라간 뒤엔 "거기 서 있다"고 생각하고 남은 길만** 잇는다. 전체 경로를 다시 적으면 이미 온 길을 또 가라는 뜻이 된다.
- 이름은 **글자 하나까지 정확**(대소문자 구분). 눈으로 확신하지 말고 `ls`로 확인하거나 **Tab 자동완성**으로 원천 차단.
- 가장 위험한 건 에러가 아니라 **에러 없이 엉뚱한 곳에 도착**하는 명령이다.
