# 🐧 Linux L5 — 프로세스 (2026-07-20 · WSL Ubuntu 24.04)

> `ps`·`&`(백그라운드)·`jobs`·`kill`·시그널(TERM/KILL)·`pkill`. OS 이론 T1(프로세스)에서 배운 개념을 *명령으로* 손에 익힘.

**개념 씨앗**: **프로세스** = 실행 중인 프로그램의 인스턴스, 각자 고유번호 **PID**. **`명령 &`** = 백그라운드 실행(터미널 안 붙잡음). **`jobs`** = 현재 쉘의 백그라운드 작업 목록(`%1`,`%2`…). **`kill PID`** = 프로세스에 **시그널** 전송, 기본 **TERM**("정리하고 종료해줘"). **`kill -9 PID`** = **KILL**(강제, 거부 불가). **`ps`** = 스냅샷 / **`top`** = 실시간.

## 예측 — `sleep 100 & ; sleep 200 & ; jobs ; kill %1 ; jobs`  → **1·3 정확, 2·4 정밀화**

- **내 예측 (그대로)**:
  > 1. 바로 다음 명령을 칠수 있다. 그 이유는 & 명령어를 사용했기에 백그라운드에서 진행되기 때문이다.
  > 2. 2줄이고, %1 & sleep / %2 & sleep
  > 3. 1번 작업번호를 종료한다. %1은 1번 작업을 얘기한다.
  > 4. %1 kill sleep / %2 & sleep
- **실제** (WSL 실행):
  ```
  (A) 두 줄 직후 → 프롬프트 바로 돌아옴 (백그라운드)         [예측 1 ✅]
  (B) 첫 jobs:
      [1]-  Running    sleep 100 &
      [2]+  Running    sleep 200 &
  kill %1
  (C) 둘째 jobs:
      [2]+  Running    sleep 200 &                         ← job 1 목록에서 사라짐
  (D) ps: 420 S sleep 200                                  ← sleep 200만 남음
  ```

| # | 판정 | 정밀화 |
|:-:|:-:|---|
| 1 | ✅ | `&` → 백그라운드, 프롬프트 즉시 반환. 정확. |
| 2 | 🟡 | 2줄·작업번호·명령 맞음. **상태 칸(`Running`) 누락.** |
| 3 | ✅ | `%1`=1번 작업, `kill %1`=종료. 정확. |
| 4 | 🟡 | job 2 유지 ✅. 근데 **끝난 job 1은 "killed 줄"로 남는 게 아니라 목록에서 완전히 제거됨.** |

- **정밀화 ①(#2)**: `jobs` 형식은 **`[작업번호] 상태 명령`**. 상태 = `Running`(실행중)·`Stopped`(Ctrl+Z로 멈춤)·`Done`/`Terminated`(끝남). 예측에서 이 **상태 칸**을 같이 봐야 "뭐가 돌고 뭐가 멈췄나"를 읽는다.
- **정밀화 ②(#4)**: bash는 종료를 한 번 알린(`[1]- Terminated`) 뒤 그 작업을 **jobs 테이블에서 제거**한다 → 둘째 `jobs`엔 살아있는 것만. `ps`도 죽은 프로세스는 안 보임.
- **교훈**: **`%작업번호`**(현재 쉘 jobs 기준)와 **PID**(시스템 전역)는 다른 번호. `kill`은 **둘 다** 받는다(`kill %1` 또는 `kill 420`). `jobs`는 상태 칸까지 읽고, 끝난 작업은 목록에서 빠진다.

## 검증 함정 — `kill sleep` (이름으로 죽이기 시도)  → **1 방향맞음·2 오답·3 미지**

- **문제**: 동료가 `kill sleep`으로 sleep 프로세스를 끄려 함. ① 동작/에러? ② 이름으로 죽이는 올바른 명령? ③ `kill -9` 습관이 왜 나쁜가?
- **내 분석 (그대로)**:
  > 1. 에러발생, 이 명령은 작업번호로 받는다.
  > 2. stop sleep
  > 3. 이건 이유를 모르겠다.
- **실제** (WSL 실행):
  ```
  kill sleep   → 종료코드 1 (실패)   에러: "arguments must be process or job IDs"
  pkill sleep  → 종료코드 0 (성공)   sleep 300·301 둘 다 이름으로 종료
  ```
- **① 정밀화**: 에러는 맞음. 단 `kill`이 받는 건 **작업번호(`%1`)만이 아니라 PID(숫자)도**. 핵심 = **이름(글자 `sleep`)은 못 받는다**(숫자 ID 또는 `%작업번호`만).
- **② ❌ 교정**: `stop sleep`은 없는 명령. 이름으로 죽이는 건 **`pkill sleep`**(이름 패턴 매칭) 또는 **`killall sleep`**(정확한 이름).
- **③ 학습 (몰랐던 것)**:

  | | `kill <PID>` (기본 **TERM**) | `kill -9 <PID>` (**KILL**) |
  |---|---|---|
  | 성격 | "정리하고 나가줘" 부탁 | "무조건 즉시 죽어" 강제 |
  | 가로채기 | 가능 → **뒷정리 코드 실행** | 불가 → 정리 못 함 |
  | 결과 | 저장·임시파일 삭제·락 해제 후 종료 | 저장 안 함·**락/임시파일 잔존**·버퍼 유실·자식 고아 |

  → `-9`는 프로세스의 **뒷정리 기회를 박탈**한다(DB·파일 쓰던 중이면 데이터 손상·락파일 잔존). **올바른 순서: `kill`(TERM) 먼저 → 안 죽는 것만 `kill -9`.**
- **교훈**: `kill`은 **숫자 ID/`%작업번호`만**(이름 ✕ → `pkill`/`killall`). **TERM(기본) 먼저, KILL(`-9`)은 최후수단** — 강제 종료는 뒷정리를 건너뛴다.

## 직접 — 프로세스 제어 명령 4종  → **✅ 4/4 힌트 없이 합격**
**스펙**: (a) `sleep 400` 백그라운드 실행 · (b) 백그라운드 작업 목록 · (c) 그 sleep을 작업번호로 종료 · (d) 이름이 `sleep`인 프로세스를 **강제(-9) 없이** 전부 종료.

```bash
# (a) 백그라운드
sleep 400 &
# (b) 작업 목록
jobs
# (c) 작업번호로 종료
kill %1
# (d) 이름으로, 기본 TERM(강제 아님)
pkill sleep
```
**실행 결과** (WSL): (a) `[1]+ Running sleep 400 &` → (b) 목록 표시 → (c) `kill %1` 후 jobs 비워짐 → (d) `pkill sleep`이 sleep 500·501 둘 다 종료(종료코드 0, 잔존 없음). **전부 정확** — 특히 (d)에서 `-9` 없이 기본 `pkill`(TERM)로 간 것 = "정중히 먼저" 원칙 즉시 적용.

## 그날의 핵심
- **`명령 &` = 백그라운드**(프롬프트 즉시 반환). **`jobs`**는 `[작업번호] 상태 명령` 형식 — **상태 칸(Running/Stopped/Done)**까지 읽는다. 끝난 작업은 목록에서 **제거**된다.
- **`kill`은 숫자 ID(PID) 또는 `%작업번호`만 받는다.** 이름으론 못 죽인다 → **`pkill`/`killall`**.
- **시그널: 기본 TERM("정리하고 종료") vs `-9` KILL("강제, 뒷정리 없음").** 습관적 `-9`는 데이터 손상·락 잔존의 원인 — **TERM 먼저, `-9`는 최후수단.**
- **`%작업번호`(쉘 로컬) ≠ PID(시스템 전역).** `kill`은 둘 다 인자로 받는다.
